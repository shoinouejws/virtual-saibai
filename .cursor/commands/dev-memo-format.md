# dev-memoの整理

`docs/dev-memo/YYYYMMDD.md` に書かれた日次メモを、後から参照・総括しやすい統一フォーマットに整理する。

## 指示されたとき

ユーザーが「整理して」「最適化して」と指示したら、**先頭から以下のフォーマット**で整理した本文を書く。**整理前のメモ本文は削除せず、ファイル末尾にそのまま残す**（区切りと見出しの付け方は「原文メモの保持」を参照）。

## 出力フォーマット

```markdown
# 開発メモ — YYYY-MM-DD

## 午前 / 午後 / タイトル（時間帯や作業単位でセクション分け）

### 目的

### 実施内容

### 成果（うまくいったこと）

### 課題・限界

### 知見・メモ

---
```

## 原文メモの保持

- 整理作業では、**ユーザーが入力したその日のメモ原文を失わない**ため、整理後の本文の**末尾**に **`---` と見出し `## 原文メモ（整理前）`（またはユーザー指定の見出し）を置き、その下に整理前のテキストを改変せず貼り付ける**。
- 既に末尾に原文があるファイルを再整理する場合は、**新しい整理本文の直後に、直前の原文ブロックを置き換える**（常に最新の「整理前スナップショット」が残るようにする）。

## ルール

- ファイル名の日付（`20260407` → `2026-04-07`）を見出しに使う。
- 時間帯（午前・午後）が明示されていればセクション分けし、`---` で区切る。
- 不明確な記述は**文脈から推測**して整理し、推測で変えた点は末尾に注記する。
- 箇条書きはそのままに、ただし冗長な表現は簡潔にする。
- 課題・比較はできるだけ**表形式**にまとめる。
- 「調査したい」「気になる」といった未整理の論点は「知見・メモ」内に**調査テーマ**として残す。
- プロンプトや使用コマンドなどの具体的な文字列はコードブロックで保持する。
- セクションに書くことがない場合はそのセクションを省略する。

## 整理後のフィードバック

フォーマット整理が完了したら、以下の観点で**客観的なコメント**をファイルの外（チャット上）に出力する。

- **記載漏れの可能性:** 成果・課題・知見のいずれかが空または薄い場合に指摘する（例:「成果セクションに具体的な結果が記載されていません」）。
- **分かりづらい表現:** 文脈だけでは意図が読み取りにくい記述を指摘し、補足・修正の候補を提示する。
- **フォローアップ推奨:** 「要検討」「気になる」等で終わっている論点を列挙し、次のアクションにつなげやすくする。

コメントは整理後に箇条書きでまとめて出力し、ファイル本文には含めない。
