# 開発メモ — 2026-04-07

## 午前

### 目的

- 作物の**常時アニメーション**用に、画像をパーツ単位で用意する。
- いちごの**収穫期**画像をパーツに分け、それぞれを動かして自然な動きを再現する。

### 実施内容

- 元画像 `strawberry-8.png` を起点に、生成 AI でパーツ抽出・質感調整を試行。
- 使用ツール: Nano Banana Pro、ChatGPT の画像生成機能（ほか、プロンプト・添付の試行）。

### 試したこと（うまくいかなかった）

| 方針 | 結果 |
|------|------|
| 日本語・英語のプロンプトで「この部分だけ」等を指示 | 意図とずれる |
| 抽出したい領域を円で囲んで添付 | 改善せず |
| 一度に取り切れない場合、中間結果を再添付して段階的に抽出を指示 | ほぼ無効 |

**典型的な失敗:** 除外したい部分が残る／残したい部分が消える／そもそも抽出できない。

### うまくいった手順（暫定）

1. 元画像を **Windows ペイント**で手作業編集（消しゴム相当で不要部分を削除）。
2. 残したい部分だけを**拡大して保存**（この時点で解像度は下がる）。
3. その画像を生成 AI に添付し、概ね次の趣旨で指示する。  
   - 実際のいちごの質感に近づける  
   - 解像度を上げてきれいにする  
   - **背景は必ず透明にする**

→ この流れで**一旦は意図に沿った出力**が得られた。

### 知見・メモ

- **「元画像から AI に直接マスク抽出させる」**より、**人手でシルエットを先に作る**方が、このワークフローでは再現性が高かった。
- 低解像度の切り抜きを渡し、**質感・解像度・透明背景を言語で指定**する形は、ツールの得意領域と噛み合いやすい可能性がある。
- 段階的プロンプト（再添付での追い抽出）は、この用途ではコストに見合わなかった。

## 午後

### 目的・流れ

- 午前で確立した暫定手順に沿い、**全パーツの画像を生成**する。
- 生成画像を組み合わせ、**収穫期（ステージ8）**のいちごを画面上で自然に見せる。

### 実施内容

- **パーツ座標調整用の画面**を実装。各パーツの x, y, z を画面上で調整し、結果をコピーしてコードへ貼り付けられるようにした。
- 静止状態で**パーツの重なり・位置が自然に見える**ところまで調整。
- **揺れアニメーション**を追加。主に `rotate` と `translateX` / `translateY`。
- 一次実装は全パーツが同時同調で動き**不自然**だったため、**パーツごとに動きの大きさに差**をつけ、見た目はかなり自然に近づけた。

### 成果

- ステージ8のいちごを、パーツ合成＋調整で表示できる状態になった。
- 差分付きの揺れで、ぱっと見は自然な動きに近い。

### 課題・限界

| 内容 | 補足 |
|------|------|
| 動きが大きいとパーツ間の**ずれ・粗**が目立つ | 各パーツは静止画のため、角度・位置の変化が大きいと破綻しやすい |
| 強めの「風で揺れる」試作では**パーツ同士のはみ出し** | 注視には耐えにくい |
| **パーツ粒度が粗い**（例: 1パーツに複数の実と茎） | 大きく速いアニメーションではリアルさが落ちる。本実装に載せる場合は要調整 |

### 知見・未整理の論点

- パーツをさらに細分化する案はあるが、**システム面のデメリット**（作物種類 × アニメーション種類で管理プロパティが肥大化する等）が気になる。
- **参照:** リアルな・多めのアニメーションを扱うアプリでは、一般的にどのような実装パターンが取られているか（調査テーマとして残す）。
